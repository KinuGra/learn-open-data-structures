## 何が嬉しい？（設計の狙い）

- FIFO（先入れ先出し）を **配列で O(1) で回す**のが目的。
- ArrayStack をFIFOに流用すると、先頭側の `add/remove` が発生して **毎回シフト → O(n)** になりがちでダメ。

## コアアイデア：循環配列（リングバッファ）

- 無限配列があるなら、先頭位置 `j` と要素数 `n` を持てばOK：
  - 要素は `a[j], a[j+1], ..., a[j+n-1]`
- 無限じゃないので、**index を mod で巻き戻す**：
  - `a[(j+k) % a.length]` に入っているとみなす
- これが「配列の先頭に戻ってくる」＝**循環配列**。

## 実装で覚えるべき状態（不変条件）

- `j` = 次に `remove()` する位置（先頭）
- `n` = 要素数
- 常に `a.length ≥ n`（満杯なら resize）

## 操作の要点

- **enqueue（add(x)）**
  - 書き込み位置は末尾：`(j + n) % a.length`
  - `a[(j+n)%len] = x; n++`
- **dequeue（remove()）**
  - 取り出しは先頭：`x = a[j]`
  - `j = (j+1) % len; n--`
- ポイント：**シフトが一切ない**ので、resize を無視すればどちらも **O(1)**。

## resize の意味（動的化）＋償却

- 配列が足りない/余りすぎのときに resize（例：`n+1 >= a.length` や `a.length >= 3n`）
- resize では、論理的な順序
  - `a[j], a[(j+1)%len], ...` を
  - `b[0..n-1]` にコピーして
  - `j = 0` に戻す
- resize 自体はコピーで **O(n)** だが、設計をちゃんとすると **m回の操作で総コスト O(m)** → 1回あたり **償却 O(1)**（ArrayStackと同じ結論）。

## 頻出ポイント

- 「キューを配列で作れ」と言われたら **リングバッファ + (index % cap)**。
- ありがちなバグはここ：
  - `(j+n) % len` の計算ミス
  - `j` と `n` の更新順
  - 空/満杯条件の取り扱い

この節の合格ラインは「**j（先頭）とn（サイズ）だけ持てば、配列でもFIFOがO(1)で実現できる**」を、自分の言葉と式（`(j+k)%len`）で言えること。
