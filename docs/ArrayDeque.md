## ArrayDeque が解決する問題

- ArrayQueue は「片端から追加・もう片端から削除」しかO(1)にならない。
- **両端で追加・削除を O(1) にしたい** → ArrayDeque。
- さらに **任意位置の add/remove も“近い端からシフト”で効率化**する。

## コア設計

- **循環配列 + `j`（先頭の論理位置）+ `n`（要素数）** は ArrayQueue と同じ。
- ただし ArrayDeque は **List インターフェースも実装**する（`get/set/add(i,x)/remove(i)` が使える）。
- 論理的な `i` 番目は常に **`a[(j + i) % a.length]`**。

## 操作の計算量（resize無視）

- **`get(i)` / `set(i, x)`**：**O(1)**（式1本）
- **`add(i, x)` / `remove(i)`**：**O(1 + min(i, n−i))**
  - 近い側の端からだけシフトするのがミソ
- **両端の操作（先頭/末尾の push/pop）**：**償却 O(1)**
  - これが ArrayDeque の名前の由来（Deque インターフェースが全部O(1)）
- 任意の m 回の add/remove に対する **resize の合計コストは O(m)**（償却O(1)）

## add(i, x) の実装ポイント（“どっちに寄せるか”が本質）

- `i < n/2`（前半に近い）
  - **`j` を1つ左に巻き戻す**（循環の先頭を1マス拡張）
  - `a[0..i-1]` を1つずつ左にコピー
- それ以外（後半に近い）
  - `a[i..n-1]` を1つずつ右にコピー
- 最後に **`a[(j+i) % len] = x`** で書き込み → `n++`
- 結果：**シフト量 ≤ min(i, n−i)**

## remove(i) の実装ポイント

- `i < n/2`：左半分を右に1つずつコピーしたあと、**`j = (j+1) % len`** で先頭を1つ進める
- それ以外：右半分を左に1つずつコピー
- `n--`、必要なら resize（`3n < a.length` で縮小）

## よく出るバグポイント

- `j - 1` のとき **`j == 0` の特別扱い**を忘れる（負にしない）
  - 安全形：`j = (j - 1 + a.length) % a.length`
- `(j + i) % a.length` の **mod を付け忘れて配列外**
- `j` を動かすタイミング（add では先に、remove では後で）
- **空/満杯の判定**：満杯は `n + 1 >= a.length` で resize、空は `n == 0`

## 一言ポイント

- 「**ArrayDequeは循環配列＋`j`・`n`の管理で、両端O(1)・任意位置はmin(i, n-i)。resize込みでも償却O(1)**」
- 「**ArrayQueueをList化したのがArrayDeque**」と言えると設計意図が伝わる。

これで 2.4 は「**なぜ両端O(1)なのか／なぜ任意位置が min(i, n-i) なのか**」を図と式で説明できれば十分です。
