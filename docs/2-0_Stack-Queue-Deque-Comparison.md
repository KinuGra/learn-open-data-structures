# Stack / Queue / Deque 重要ポイントまとめ

## 一発判定

- **Stack (LIFO)**：巻き戻し・入れ子・DFS
- **Queue (FIFO)**：到着順・BFS・最短経路（重みなし）
- **Deque**：両端操作・Sliding Window・LRU
- **Priority Queue**：優先度順に取り出せる派生型（後述で軽く紹介）

## 各構造の要点

### Stack（後入れ先出し）

- 操作：`push`/`pop`/`top` すべて **O(1)**
- 用途：
  - **DFS（深さ優先探索）**：今いるノードから「行けるところまで潜って、行き止まりなら戻る」を実現するのにスタックが自然に使える
  - **括弧マッチ・式評価**：開き括弧を積んで、閉じ括弧で一致確認。逆ポーランド記法の電卓もこれ
  - **Undo/Redo、ブラウザの戻る**：直前の状態を積んでおいて巻き戻す
  - **Monotonic Stack**：「次に自分より大きい要素はどれ？」を配列を1回なめるだけで O(n) で求められる（ヒストグラム最大長方形など）
  - **再帰の代替**：再帰が深すぎてスタックオーバーフローしそうなときに、自前のスタックで書き換える
- 弱点：末尾しか触れない／FIFOしたいのに使うとバグる

### Queue（先入れ先出し）

- 操作：`enqueue`/`dequeue` **O(1)**（**循環配列**で実装が必須）
- 用途：
  - **BFS（幅優先探索）**：浅い層から順に広げるのでキューと相性が良い。**重みなしグラフの最短経路**はBFSでO(V+E)
  - **タスクキュー / メッセージキュー**：到着順に処理（印刷ジョブ、Kafka、SQS など）
  - **Webサーバのリクエスト処理**：先に来たリクエストから順にさばく
  - **生産者・消費者パターン**：スレッド間でデータを受け渡すバッファ
- 弱点：途中要素にアクセス不可／単純配列だと先頭シフトで O(n)

### Deque（両端キュー、「デック」）

- 両端の `pushFront/pushBack/popFront/popBack` がすべて **O(1)**
- 用途：
  - **Sliding Window 最大/最小（Monotonic Deque）**：長さ k の窓を1つずつずらしながら、その中の最大値（最小値）を毎回 O(1) で取り出す。全体で O(n)（LeetCode 239 が代表例）
  - **LRUキャッシュ**：最近使ったものを先頭に動かし、溢れたら末尾を捨てる。Deque + HashMap で O(1) 実装
  - **パリンドローム判定**：両端から1文字ずつ取り出して比較
  - **Stack/Queue の汎用代替**：C++ `std::deque`、Python `collections.deque` のように、両方の役割を1つでこなせる
- 弱点：実装がやや複雑、中央挿入は O(n)

### Priority Queue（軽い紹介）

- 「**入れた順ではなく、優先度の高いものから取り出せる**」キュー。内部はヒープで実装され、`push`/`pop` は O(log n)。Dijkstra や TopK などで頻出。
- ここでの **「優先度」** とは、要素ごとに付ける「取り出される順番を決める基準値」のこと。
  - 例：病院のトリアージ＝症状の重さ／タスク＝締切や重要度／Dijkstra＝スタートからの距離／OS スケジューラ＝プロセスの priority 値
  - 何を優先度にするかは問題ごとに自分で決める（距離・コスト・締切など）

## 選び方の軸

1. **順序の向き**：最後 → Stack ／ 先に来た順 → Queue 系
2. **両端を触る？** → Deque
3. **優先度がある？** → Priority Queue
